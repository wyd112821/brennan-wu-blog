《技术团队管理者的第一堂管理课》读书笔记

# 技术团队管理者的第一堂管理课

# 前言

技术团队管理者的核心工作是带人、做事、建体系。人带不好，事就做不成，而带人高效地做事，终极解决方案则是建立规范化的流程体系。本书按照从带人到做事，再到建体系的顺序进行介绍，将技术团队管理者的核心工作从这3个维度拆解成8个模块。

下图是这3个维度和8个模块的框架图。

第一个维度是带人，分别用团队建设、人才管理和文化塑造这3章进行描述，揭示技术团队管理者带好人必须要有度量，能成就他人；

第二个维度是做事，分别用项目管理和效率管理这2章进行描述，展示做事的核心要讲求方法，注重效率；

第三个维度是建体系，分别用绩效管理、流程建设和规范管理这3章进行描述，它的核心是控制和公开透明。

这3个维度可以有效地帮助技术团队管理者从0到1打造出一支强大的团队。

# 第1章　团队建设

# 1.1　新官上任

新上任的技术团队管理者由于身份和职责的快速变化，往往会水土不服，很难适应新岗位，其原因归结起来有以下3种：

  • 思维难变：从技术岗位突然到管理岗位，缺少培训和指导，更没有过渡期，一直用技术人员思维带队做事，每天为各种事情焦头烂额。
  • 排挤他人：有些新上任的技术团队管理者为稳定地位，排挤资历老、不服管的员工，而任用自己信任和听话的员工。
  • 自吹自擂：新上任时信心满满、自吹自擂，干活时能力有限、水土不服，最终以项目失败而告终。

作为一个新上任的技术团队管理者，如何才能取得团队和领导的信任，务实地统领全局，整体规划，分步实施，短期内做出业绩呢？具体做法如图所示。

# 1.2　角色认知

技术人员升职为技术团队管理者后，其实工作内容的前后变化还是很大的，在时间、目标、职责、精力和关注点等方面都会截然不同。具体表现如下：

  1. 职责不同：技术团队管理者更要对整个团队负责，而不像技术人员一样只负责代码开发，不考虑其他团队成员的代码质量和效率。
  2. 时间不同：技术人员有整段时间可以做事，只要按计划执行，就可以完成任务，而技术团队管理者更多的是在碎片化的时间里带团队做事，计划往往赶不上变化。
  3. 问题不同：技术人员在工作中只需要解决自己的技术问题，就可以确保工作成果，而技术团队管理者每天要花费大量时间处理下属的问题，才能扫清项目执行中的障碍。
  4. 沟通不同：技术人员遇到问题只需向上沟通，而技术团队管理者不仅要向上沟通，还要向下沟通和平级沟通。
  5. 质量不同：技术人员只对自己的代码质量负责，而技术团队管理者要考虑项目整体质量，才能确保结果有较高的质量。
  6. 思考不同：技术人员只需思考如何提升自己的技术能力，而技术团队管理者要考虑项目的进度、质量、成本等诸多因素。
  7. 关注不同：技术人员平时关注技术的时间较多，而技术团队管理者为了提升团队整体的效率，平时关注人和事的时间较多。
  8. 目标不同：技术人员的目标一般是事情怎么做，重点是过程，而技术团队管理者的目标是找谁做，重点是结果。

上面的8个不同点，每一点都需要新上任的技术团队管理者学会“管人”，职位的变化导致工作内容完全转变，这些新的内容需要技术团队管理者去适应，并清楚具体的工作职责。

# 第2章　人才管理

人才是一个企业的核心竞争力，企业对于人才的需求永远都是第一位的。对企业来讲，招聘到合适的人才，建立完善的人才梯队，是企业发展的重要基础。

# 2.1　招聘

为了提高招聘效率，让招聘工作责任到人，需要建立一套完善的招聘流程，如图所示。

决策时，除了依据求职者专业能力的考察得分，同时还要考虑求职者的能力和团队互补情况、求职者对公司和项目的兴趣程度、求职者与团队文化的匹配度，以及求职者的可塑性等。

作为技术团队管理者，最开心的事情是招到比自己厉害的人，这样整个团队效率就可以提升，工作中又多了一个好搭档，团队更多了一个学习榜样。

# 2.2　解雇

从某种意义上讲，一个没有亲自解雇过下属的技术团队管理者是没有资格独立进行招聘面试的，因为一个技术团队管理者如果没有经历过解雇下属，就很难体会到解雇的难处，进而更不知道招聘时需要多么谨慎。

对于能力不足的员工被解雇，其实很大一部分责任来自技术团队管理者，是技术团队管理者选人和育人的能力不足导致的。对于能力不足的员工，技术团队管理者可以从下图所示的几项进行自检。

每个技术团队都有自己的一套选人、育人和留人的方法，技术团队管理者要克服自身的困难，站在部门或公司层面上看问题，其实部门和公司发展壮大的过程中人员更换是常态，也是必然，以平常心看待就可以了。

# 2.3　培养

对技术人员来讲有两样非常重要的能力，一是学习能力，二是反思能力，学习能力决定技术人员的成长速度，而反思能力则决定技术人员学习的速度。

对技术团队管理者而言，有一项比较重要的能力——快速提高团队成员的学习能力的能力。学习能力提高后，团队的战斗力会非常惊人。

对于学习，我们需要找到大脑的认知和学习规律，那就是反思性学习和间隔性学习。掌握了这两条规律，学习效率就会高很多，这两条规律适用于任何人。

技术人员日复一日机械式编码，大脑其实是处于“偷懒”状态，会缺乏思考的过程，而对所学的知识不进行深入的思考，到应用层面上的效果会打很大的折扣。进行及时的反思是一种非常高效的学习方式，它有助于加深大脑对所学知识的理解和记忆，也符合大脑的认知规律。

很多技术人员喜欢集中一段时间学习一门新技术，因为这样很快能入门，所以误以为集中时间学习效率更高，其实这是假象。虽然集中时间学习可以在短期内快速地获得收益，但是遗忘速度非常快，长期收益率非常低。而间歇性学习虽然短期收益小，但是从长期来看收益却很大。

技术团队管理者要以身作则，引导团队坚持学习，并教会团队一套学习方法。

技术团队管理者要帮助团队成员多练就独立思考能力和技术方案输出这两件事情。

有了足够的知识储备和思考模型，才能有独特的看问题的角度，才有可能给出客观中立的评价，从获取信息海量再到分析这个过程不易，有时甚至逆向而行，但成长路上从来都不会轻松，这个过程很漫长，也恰恰因为过程漫长，持续践行才更有价值。

技术团队管理者组织团队定期做技术方案的讨论、选型和PK是训练技术人员逻辑思考能力的一种非常高效的方式，也是学习和实战的综合演练。通过输出方案可以检测知识存量、思考力和解决问题的能力，知识就是在思考到实践的过程中产生的，通过实践可以帮助团队具有更强的思考力。

# 第3章　文化塑造

团队文化是团队协作中形成的一种潜意识文化，是团队观念和持续践行的结果，它存在于团队成员的心里，融入日常的点点滴滴，可以有效地统一团队意识，管理团队情绪，约束团队行为，提高团队效率，它可能有缺陷，但每个团队的文化却是量身定制的。文化践行的过程中，需要技术团队管理者有很强的感染力和执行力，带领团队脚踏实地地做好每一个环节，把细节落到实处，让文化能够刻在团队成员心里、落实在行动中，持续推动团队良性发展，直至变成团队的工作习惯和做事准则。

那么，作为技术团队管理者如何才能有效地打造优秀的团队文化呢？

# 3.1　守一个“诚”字

技术团队管理者在工作中塑造团队坦诚文化，可以从实事求是、树立榜样、不包庇错误和学会变通这4个方面入手，如图所示。

技术团队管理者要合理评估团队成员的技术能力，为他们安排合适的工作，不强迫团队成员做超出其能力范围的事；“实事求是”地尊重技术，评估方案，通过技术方案的PK，让团队成员学习到不同的设计思路和技术实现。

技术团队管理者平时要以身作则做团队的表率，身教优于言传。首先，做人、做事要有原则和底线，建立自己的行为准则，才会得到别人的尊重；其次，要主动承担团队重要责任，深入一线解决技术难点，带领团队披荆斩棘，而不是在后方“指点江山”；再次，不八卦别人的工作和生活，要做到不私下讨论团队成员的优缺点，不分享别人的私密事情；最后，委婉地指出团队成员的问题，指出别人的问题也是一种坦诚的表现，对方当时心理上可能无法接受，但是只要是真心为对方好，对方迟早会理解。

技术团队管理者不仅需要表扬和激励团队成员，还需要批评和指正团队成员的错误，团队成员犯错误后予以改正才能快速成长。不包庇错误，客观公正帮助对方成长，避免团队其他成员犯同样的错误

技术团队管理者只需要告诉他们问题所在，应该怎么做，如何做才能更加有效率。从结果出发，帮助他们成长，才能更让人信服。

# 3.2　构建学习型组织

技术团队管理者把自己的团队构建成学习型组织，可以有效地激发团队的创造性思维，培养团队的学习氛围，帮助团队成员持续学习和分享知识。而分享又是一种非常有效的知识整合的方式，是对分享者所学知识的实战演练。工作中技术团队管理者要注重团队成员的知识共享。

对于构建学习型组织，首要就是要营造轻松的学习氛围，让团队成员爱上学习和分享。

研发部门是一个技术型团队，技术的升级迭代速度非常快，需要技术人员不断学习新知识，技术团队管理者组织团队通过读书分享的方式帮助团队成员学习和成长是不错的办法。

建立一套wiki知识系统来收集团队工作中积累的知识，把工作中涉及的文档包括但不限于产品需求、技术方案、测试用例、经验总结、分享经验、流程规范等统一在wiki知识系统中编写和管理，有利于团队学习和共享，更便于培训和总结。

团队成员坚持分享能帮助团队持续成长，技术团队管理者要加强对内部分享的重视，要求分享的内容要有价值、有针对性和实用性，对于优秀的分享者可以参与外部分享或推荐给企业做全员分享。

# 3.3　在复盘中成长

技术团队管理者在日常的工作中就要增加团队的复盘意识，通过一些问题提升团队的反思和总结能力，例如，做过项目的不足之处有哪些？如何改进？项目执行中的关键点有哪些？如果从头再来一次，如何能做得更好？一个有复盘意识的人是一个喜欢反思的人，因为复盘的本质是思之再思的过程，也是团队能力提升的关键策略。

通过复盘也可以知道团队的优势和劣势，便于对项目风险进行评估，还可以磨炼团队成员的细节执行能力，简单来讲，复盘可以弱化劣势，强化优势，明确团队的价值和目标。

为了能让团队的思考都聚焦到核心的问题上，可以按照如下4个步骤复盘，如图所示。

  • 回顾目标：对目标进行回顾，将当初制定的目标和实际结果进行比对和评估，找出具体的差异和问题。
  • 还原过程：还原执行过程，分析当时的操作流程和执行细节，找到执行中的关键影响因素，通过对比和分析数据，找出根本问题。
  • 分析原因：通过分析执行中的关键影响因素和数据，对问题进行反思，找到问题的根源，避免重蹈覆辙。
  • 总结经验：把执行中的经验总结出来，应用到工作中，对不足之处制订改进计划。

# 3.4　新人指南

新人加入团队后，需要快速熟悉部门业务和工作内容。此时团队需要一份新人指南清单，以帮助新人快速开展工作。要获取指南清单的每一项内容可跳转详情页，便于新人了解具体信息。新人指南清单的内容包括产品篇、账号篇、环境篇、流程篇、规范篇、制度篇、技术篇、安全篇、工具篇和经验篇。

新人融入团队的速度靠的就是团队中点点滴滴的细节，这些细节的完善需要技术团队管理者持续思考和优化团队工作流程，才能让团队成员更有归属感。

# 3.5　新人访谈

新人在入职后不久需要技术团队管理者对新人进行一次访谈，了解新人对团队的感受及价值观的认识，鼓励新人说出自己的真实感受，包括工作中的问题、个人的顾虑、遇到的困难等，这些真实的表达能帮助他们更快地适应环境，解决工作中的问题，调整工作心态和规划好长期工作。访谈内容包括主要工作、工作复盘和工作计划，如图所示。

# 3.6　周会

周会的目的是同步工作进度、思考和计划。周会内容至少包括3部分：

  • 上周主要工作：汇报上周完成的主要工作、工作完成的基本情况、遇到的阻碍及需要协助的地方，汇报的内容要有重点，语言简洁。
  • 上周工作复盘：对上周工作中各项事务的思考和总结，需要整理成经验知识或操作指南，及时记录到wiki知识系统。
  • 本周工作计划：本周的主要工作，包括参与的项目、执行关键点、协调事项、跟进问题、团队协作等。

周会的时间不宜过长，要有一定的纪律，不能进行无关话题的讨论，同时需要输出周会纪要，用于确定跟进事项和整理周会内容，同时也便于追踪问题。

# 3.7　坦然面对失败

技术团队管理者如果在工作中一直一帆风顺，从未失败过，那么大概率是在工作岗位上也没取得过什么成绩，因为机会通常和风险并存，如果害怕失败，惧怕风险，机会自然也就会与他擦肩而过。简单重复的工作并不能使人在工作中提升能力，技术团队管理者如果没有试错和变革的勇气，就很难给公司和团队创造价值。

要想在工作中无惧失败，不断试错，并非易事，面对失败时需要有一些策略。

首先，要降低失败的成本。其次，要改变对失败的认识。

技术团队管理者尚且如此，团队成员自然也会效仿。所以技术团队管理者不仅需要克服自身对失败的恐惧，还要鼓励团队成员不断试错，团队成员的每一次试错都是在探索和创新。

当一次失败产生，尤其是面对比较重大的问题时，技术团队管理者首先要弄清楚问题产生的原因，避免过早对问题进行定型并追究责任。技术团队管理者面对失败的一言一行都会让团队重新认识失败，应减少失败带来的困惑，提升团队面对失败的勇气。

# 3.8　直言不讳

作为一个合格的技术团队管理者，应该接受团队成员的不同声音，即使他们的建议不靠谱，也要认真聆听并告知不采纳的原因。

当然对于下属的建议并不是都靠谱，技术团队管理者可以从4个维度考量所提的建议是否有价值：

  • 一是提出的建议是否经过充分的准备，没有调研就没有发言权；
  • 二是提出的建议对事不对人，聚焦事情的目标和结果；
  • 三是对建议进行逻辑推导，判断考虑是否周全；四是自己是否愿意尝试，推进方案落地执行。

一个团队最好能直言不讳，直言不讳能帮助技术团队管理者更细致地思考问题，也能带动团队表达真实的想法。

# 第4章　项目管理

与传统软件公司相比，大多数互联网公司并没有单独设立项目经理职位，而是由技术团队管理者承担项目管理工作，但项目管理不是一项简单的工作，需要系统性地考虑项目的诸多因素，例如制度设计、质量监控、版本控制、环境维护、进度控制以及项目相关者的利益协调与分配等事项。这需要技术团队管理者有较强的综合能力，不能只关注局部问题，忽视项目的整体进展，而要全局思考项目的方方面面。

# 4.1　项目启动

为了让团队清晰地知道项目目标，有计划和目的地开展工作，同时避免在项目实施过程中犯一些低级错误，就需要有一个好的项目启动，好的项目启动能增强团队的信心，为项目的后续开展减少阻碍并获得更多的支持。

对于项目启动环节，有3部分工作，分别是确定项目目标、确定项目人员和项目启动会。

项目目标的制定需要遵循SMART原则，具体表现在：

  • 具体性。制定的目标要有详细的指标描述，目标可以简洁到用一句话来概括，目标是否明确，关系到项目最终的成败。
  • 可衡量性。制定的目标不仅要明确，目标的交付成果还必须是可衡量的，有明确的数据指标来衡量目标的完成效果。
  • 可实现性。目标制定要充分考虑团队能力的匹配度，实现的目标要具有挑战性，同时目标的难度又不至于让团队失去信心。
  • 相关性。目标的实现要考虑是否能同时满足多个业务，这样目标实现后价值更大，实现一个目标就可以解决多个问题。
  • 时限性。要考虑在限定的时间内完成目标。

技术团队管理者还要给目标赋予意义。需要告诉团队为什么要做这个项目，对技术人员来讲，高质量地把项目交付上线不仅意味着完成工作职责，给自己带来成就感，同时还能带来技术进步和成就自己的可能性，这些都是项目目标带给团队的意义。

制定的目标要共享，确保信息公开透明，目标一旦公开团队就没有退路，进而做事就会更有动力，同时公开的目标能让业务部门更加了解技术团队的工作情况，进而获取资源上的帮助和排期上的认可。

典型的互联网项目团队人员结构包括项目负责人、产品经理、设计师、前端工程师、架构师、后端工程师、测试工程师和运维工程师 对于确定哪些人参与项目开发，可以用竞选的方式，在部门内设立竞选机制，把技术团队划分成多个项目组，每个项目组之间横向PK和学习。

人员确定后的下一步工作就是确定信息同步机制，保障项目组成员间的信息透明，信息透明的情况下共享团队的协作效率才会更高。

对于项目启动会，要求项目全体人员参加，包括其他与项目有利益关系的编外人员。会议内容如下：

  • 价值说明：说明项目的价值和意义，帮助团队了解项目的整体情况。
  • 确定目标：确定项目目标及制订项目规划，让团队知道项目如何开展。
  • 时间节点：确定项目的里程碑时间节点，让团队知道在各个时间节点需要输出的关键成果。
  • 职责分工：确定项目团队成员的职责及其分工，让团队职责清晰。
  • 风险应对：确定项目资源投入及风险应对策略，让团队提前做预案。

# 4.2　需求管理

技术团队管理者在确定需求范围的同时，还要学会理解和转化需求，把业务方表述的需求转化为真实场景需求，再把真实的场景需求转化为功能需求。

对于项目工期的估算，需要考虑包括项目需求、任务前后顺序和依赖、资源投入、人员能力、延期风险、外部影响和交付质量等因素。对于交付质量较高、难度较大、协调沟通事项较多的项目，估算工期时要考虑预留一部分弹性时间，尽量降低项目延期的风险。

# 4.3　进度管理

我们需要先制订项目进度计划。制订的进度计划要细化到天，保障每天的工作可量化，每周的工作可验收，依照进度计划严格执行。

每日站立会在敏捷开发中非常重要，通过站立会可以保障信息透明，让团队成员快速了解各自的具体工作和进度，便于全面把控项目进度。

进度看板秉承可视化管理的理念，用于过程管理，保障研发过程能最大限度地可视化呈现。

项目管理者要学会高效的沟通。包括需求沟通、紧急沟通、会议沟通和跨部门沟通。

对于变更的需求，不论大小，都需要严格按照变更流程进行管理，且变更的内容不能通过口头传达，而需要得到对方的确认，在需求变更确认的内容中要明确变更的影响因素，包括时间、成本、范围、风险等，这样才能够让项目参与方慎重地对待需求的变更。然后，待双方意见一致，签署需求变更确认文档后，需要通过电子邮件通知全体项目人员，保证信息的同步和流畅。

# 4.4　风险管理

要进行项目风险识别，首先要设立项目风险委员会，风险委员会的成员来自项目组的核心人员和外部的专家，他们有丰富的项目经验，能够确保项目中的大部分风险第一时间被识别，从而提前做好风险预案并控制风险。

对于识别出的风险，除了及时登记在册，还需要对风险进行评估，并输出风险管理计划，对于风险的评估可以从发生概率、产生影响、关联因素和发生时间4个维度进行分析。

对于风险的控制，首先要对评估出的风险查找问题根源，找到问题源头才是解决问题的根本之道。同时，在风险控制上还需要建立预警机制。另外，对于识别和评估出的风险，要明确责任人，及时跟踪和记录风险信息，确保风险在发生的第一时间能得到处理，而不会因为风险无人负责而发生事故。

# 4.5　质量管理

技术团队管理者要重视代码质量，带领团队严把质量关，增强质量意识，在项目执行中落实每一步行动，通过长期的践行才能最终形成质量文化。

技术团队需要搭建一套质量管理体系来提高项目质量，质量体系的搭建可以从3个部分入手，包括预防准备、过程管控和质量平台集成，如图所示。

# 第5章　效率管理

# 5.1　提高时间质量

对技术团队管理者来说，工作时间一定是碎片化的且又需要随时在线，要想有一段时间专注地做事情而不被打扰，很难！

那有没有什么办法能够在有限的碎片化时间内快速提高工作效率，一天做别人几天的工作呢？下面有3个方法可以试一试。

  1. 公开目标：公开的事情会受到监督，基本没有退路，变成了不得不做的事情，有了这种压力，自然会逼着自己做出结果，完成事情的过程中时间的利用效率也会更高。
  2. 时间日志：时间日志是一种时间管理方法，把对时间的管理分为3个阶段，分别是记录、分析和调整，如图所示。
  3. 深度工作：要想在碎片化的时间里深度工作，首先需要先找一个适宜的环境，给自己腾出一段时间集中工作，对于中途临时插进来的工作，定时处理就可以，除此之外，还需要提高工作中的时间质量，使工作更加高效，如图所示。

# 5.2　高效地学习和工作

我们生活在今天这个信息极度膨胀的时代，海量的信息通过互联网和书籍传播，每个人都能轻而易举地获得。知识并不匮乏，甚至有点儿泛滥，面对海量的知识我们必须高效地进行学习和工作。

想要高效地获取知识，首先要有一个好问题，然后带着这个问题有针对性地找资料认真研究学习，通过一个问题掌握其相关的知识点，再通过学到的这些知识点延伸学习新的知识，这样就可以花很少的时间做到既解决问题，又获取有价值的内容，效率会提升好几倍。

技术人员或技术管理者平时要多和同行业的牛人交流，以获取更多有价值的信息。

技术变化日新月异，新技术和新框架每天都在更新，技术团队管理者不能仅凭借自身实践经验带团队做项目，而要提高自身获取知识的速度和深度。技术团队管理者是团队的舵手，知识的深度和获取速度会影响团队的方向和效率。

当我们发现自己每天的工作和生活陷入了一个惯性的状态时，就需要打破惯性思维，从更高的角度来看待这些问题，不断反思自己的工作和生活，保持敏锐的观察力和足够的好奇心，不要只是日复一日地惯性工作，留一点时间给自己思考，在工作实践中不断反思学习。深度思考能够让你更加高效地工作，帮助你节省更多的时间。

技术人员和管理者，其实本质在干同样的事。技术人员依靠个人能力解决问题，而技术团队管理者是带领一群人解决难题，都是解决问题，只不过路径不同罢了。成为管理者可以突破个人价值的天花板，站在更高的层级看问题，看到的问题会更全面，发挥的效能也更大。

清楚地了解自己要什么并为之付出努力，这是一切有效积累的前提，它能扫除你内心的不安与恐惧，渐渐地就会对当前工作产生兴趣，进而会更加热爱，只等待一个市场机会到来，你就会成为这个行业中优秀的人。

技术团队管理者需要有处理事务的综合能力，对下，要有用人、管事、出结果的能力，带领团队成长并为目标负责；对上，要有总结汇报能力，并把握公司整体技术发展方向；对自我，需要不断加强内在修炼，提升技术能力和自身的综合素质；横向，要有出色的协调和沟通能力，提升横向协作效率；对业务，不能仅局限于项目功能，而是要站在商业角度，深度融合技术和业务，通过技术驱动业务；对技术，要持续学习新技术，解决系统核心技术问题，提高团队整体技术水平。

# 5.3　精通技术，更要懂业务

作为一位技术团队管理者，必须让别人知道自己的技术实力，实力代表着信任，同时也会争取到更多的权利，有了信任和权利会减少很多阻碍，工作推进也会更加顺利。同时，技术团队管理者的技术实力主导着整个部门的技术方向，搭建合理的技术架构，能让团队在开发中事半功倍，他们的能力的强弱决定了团队会走多少弯路。

但要注意的是，技术团队管理者虽然要精通技术，但是更要懂业务，即能站在业务的角度思考技术问题，精准地切中需求，真正解决实际业务问题。评估技术方案时，不能带有“技术为王”的思想，因为技术服务于业务，应更多地从业务角度对技术进行架构设计，避免盲目地追逐技术的高大上。

# 5.4　沟通管理

作为技术团队管理者，向上汇报非常重要，如果能高效地汇报工作进展，让领导觉得办事靠谱，自然就能获得更多的工作支持，工作的开展也会更顺利。

向上汇报，首先需要明确目标，即向上汇报的重点事项是什么，认真分析领导听完汇报后提出问题的真正意图。其次，为了让汇报能抓住重点，节省双方的沟通时间，需要对信息进行整理和汇总。最后，汇报时最后用结论+问题+建议的形式汇报。

技术团队管理者搞定横向协作事务的能力显得非常重要。在横向协作上，首先需要制定合理的跨部门协作规范、制度和流程，确定做事的协作机制，按照流程规范执行工作，这样职责才会更加明确，除了制定规范流程，还需要让业务团队承担需求开发的成本。

对于技术部门和业务部门之间横向协作效率低的情况，技术团队管理者要深入一线找到问题的根源，避免资源不必要的浪费，帮助推动业务和技术的高效合作。

沟通是一门学问，需要不断地锤炼，当出现沟通障碍时，我们要先从自身找问题，换位思考，多体谅别人的难处，这样可以帮助你扫清很多阻碍。 真诚待人，努力工作，提升能力，这些都会被别人看得到，保持一颗正直的心，别人自然会尊重你。

识人的能力对技术团队管理者来说非常重要，对于识人，很多技术团队管理者喜欢在和下属的沟通中，通过下属的沟通能力用感觉评判下属的能力强弱。 技术团队管理者不能简单地用一个下属是否“会说话”来区分下属的能力，这样容易被自己的主观情绪影响，做出对下属能力的错误判断，最终导致团队变成“说得好听，做得难看”，了解一个人，不是看他说了什么，而是看他思考了什么，做了什么。

# 第6章　绩效管理

# 6.1　认知绩效考核

绩效考核的主要作用包括以下4点：

  1. 利益驱动，激发潜能。绩效结果直接决定薪资收入，通过薪资收入激励员工，给员工树立公平的竞争环境，员工所有想要的都可以靠自己的努力和奋斗得到。员工不断拼搏，企业才能获得快速发展和壮大。
  2. 达成目标，提升业绩。把企业目标和员工行为联系起来，通过把企业目标分解成绩效指标，达到激励员工行为的目的，可以有效地帮助企业改善和提升业绩。
  3. 价值评估，衡量贡献。绩效考核是对员工创造的价值进行评估，衡量员工做出的贡献，进而帮助企业发现优秀人才，这种考核机制促进了企业的健康发展，也帮助员工不断完善自我。
  4. 挖掘问题，优化改进。绩效考核是一个目标设定、执行、改进，不断发现问题，并再优化的过程，是企业进化的重要手段。

# 6.2　传统绩效考核的弊端

  1. 产出价值量化难：评估技术人员的工作成果价值的关联因素较多，无法设定统一的考核标准。技术人员平时需要持续钻研和思考，才能出色地完成一些挑战性的工作，所以，对于他们的绩效结果的考核，需要起到激励创新精神和培养创造能力的作用，这些创新精神和创造能力，如果通过一些可量化的指标进行考核，最终往往不会理想，甚至事与愿违，怨声载道。
  2. 业务融入量化难：业务融入度是一个很难量化的指标，跨部门合作这面“墙”一直影响技术和业务的深度融入和协作，成为很多企业的痛。如何设定合理的绩效考核指标让企业信息透明共享、目标清晰一致，发挥技术人员最大效能，一直是技术团队管理者要持续思考的问题。

# 6.3　敏捷的绩效考核

那么绩效考核的核心到底是什么呢？其实考核的核心只有一个——构建自驱的内核文化，增进部门合作仪式感，以关键技术驱动业务增长。 物质激励不一定对每个技术人员都起作用，驱动和激励优秀的技术人员工作的往往是自我价值的实现。所以，要改变常规绩效考核的方式，打造出高绩效的敏捷团队，技术团队管理者可以把绩效考核分为下面3个部分：责任、价值和成长。

  1. 责任：技术团队管理者要严格检查工作完成情况，要求每件事情做到“极致”，通过每一件事情的完成结果关联团队成员的“责任”。尽责做事能帮助团队提升解决问题的效率，磨炼团队的韧性，同时尊重岗位工作，在持续解决问题中获得做事的动力。分析问题靠的是智慧，解决问题靠的是责任，很多时候责任比智慧更重要。技术团队管理者通过要求团队尽责做好每一件事情，帮助团队成员建立责任意识，记录价值贡献，可视化每个人的成长，让团队在工作中学习进步，在尽责解决问题中锻炼成长。
  2. 价值：任何一个人都想在团队中体现自己的价值，一个组织如果能持续给员工展现价值的机会，这个组织的离职率一定会很低。对一个技术人员来讲，价值体现在： a. 技术攻关：包括质量控制、技术分享、规范建设和自研项目。 b. 组织共建：包括人才管理、目标激励和文化建设。 c. 协作创新：包括横向协作、进度管理和技术驱动。
  3. 成长：成长对技术人员来说至关重要，技术团队管理者要帮助每个技术人员制订成长计划，并在关键的时间节点检验成果。要帮助团队成员制订一份成长计划，首先，知道对方当前的能力阻碍；其次，客观评估预期目标；再次，时间周期内设定关键动作；最后，考核指标要可衡量。所以，一份科学的成长计划应该包括能力阻碍、预期目标、关键动作和考核指标这4个方面。

# 第8章　规范管理

有经验的技术团队管理者会从规范制度入手，用标准的输入动作去影响团队的执行结果，让团队从中收获到成功的动力，由这种及时反馈的动力来推动团队前行，这样才能最终固化成团队的执行习惯。

# 8.1　认知规范

规范的价值主要体现在快速上手、减少错误、评价依据、约束行为和提高效率这5个方面，如图所示。

提升团队效率的核心还是要让团队高效地协作，协作中的损耗越小，信息的传输和理解效率就会越高，这也是技术团队管理者制定各种制度、规范和流程的初衷。

技术团队的工作规范是经过团队不断试错和完善而形成的，这并非一朝一夕的事情。规范的形成会经历初步规范、问题诊断和优化改进这3个阶段，如图所示。

# 8.2　新人指导规范

技术团队管理者必须摆脱路径依赖，不能对既定的团队流程习以为常，要不断优化流程和规范，考虑如何让新加入的小伙伴快速地熟悉和开展工作。整个新人指导计划分为4个阶段。

  1. 第一个阶段主要是掌握基础工作内容，包括了解团队、熟悉流程、开通权限和学习wiki知识系统的内容，掌握常用工具、流程规范、应用配置和账号权限，掌握日常项目代码、线上系统、应用配置和网络架构，熟悉数据备份、配置变更、项目脚本和发版事项，熟悉脚本和日志，学习排查经验和持续集成工具，熟悉上线规范和周访谈。
  2. 第二个阶段主要是代码编写与服务监控，包括完成具体业务模块的开发，完成业务模块的单元测试，编写业务日志收集规则，深入理解监控系统的各项指标和数据，优化系统配置和脚本以及参与服务的日常监测、维护、告警等工作。
  3. 第三个阶段主要是参与工作实战，具体包括跟踪一次项目上线，加快熟悉流程规则，参与服务端开发和架构工作，根据业务需求制定技术解决方案，负责问题排查及应急问题的处理工作，在主管的指导下完成服务器集群部署、管理和监控工作，检查服务的性能瓶颈和安全策略以及对现有研发工作的优化提出改进建议。
  4. 第四个阶段主要是独立负责日常工作，具体包括制定数据备份与恢复策略，参与技术体系的搭建，保障服务稳定性和安全性，编写系统常用的组件和公共服务，参与日常的系统发版、监控、升级和调优工作，参与制定团队的流程、制度、规范和工作计划，参与团队文档化和自动化工作及月度工作总结复盘。

新人指导计划的4个阶段的时长通常为1个月，每个阶段的时长为1周。技术团队管理者根据新人在每个阶段的表现对新人做出评估，指导计划进行中新人的表现可为新人试用期后的去留决策提供参考的依据。

# 结语

本书从带人、做事和建体系3个维度梳理了技术团队管理者与团队之间的行为和关系，我们从团队建设开始，讲到人才管理、文化塑造、项目管理、效率管理、绩效管理、流程建设和规范管理。拆解完这几个部分你会觉得团队管理并不难。

带人要站在团队立场思考问题，有度量，做团队的伙伴，成就团队伙伴。做事要有原则，讲效率，把控细节，过程公开透明，通过敏捷高效的组织形式和流程规范做到职责共享，为结果负责。这样技术团队管理者才能赢得团队的信任，才能带领更多的人。

其实“带人”“做事”“建体系”是每个技术团队管理者必须要学会的3项能力，这3项能力也是有递进关系的，“带人”是比较基础的部分，“做事”是中级管理者花时间比较多的地方，而“建体系”则属于管理的内核。

团队管理说到底还是要“建体系”，“带人”和“做事”都是建体系时要考虑的部分，体系建好了，人才能用好，事情才能做成，才会有结果。刚转型的技术团队管理者可以先从设定带人做事、设定规则入手慢慢建立起团队文化，由浅入深地对团队进行管理。

从规则到文化是自然形成的过程，基于这两点，我们可以感受到所有的文化都是有规则的，从规则出发才能提高团队协作效能，但做好这两点其实并不容易，建立规则需要技术团队管理者从团队的日常琐碎的工作中快速识别问题，并找到合理的改进方案，这需要有很强的洞察能力。而文化塑造就更不容易，除了要求有洞察力和感知能力，还要有识人和驭人的能力，但也不要气馁，技术团队管理者如果把本书的内容一点点应用到实践中，技术团队管理就会一点点清晰起来。
